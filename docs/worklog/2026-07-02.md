## 1. 근본 원인 (가장 중요)

지문 인증·홈 로딩이 느렸던 **진짜 원인은 Vercel 함수 실행 리전이 미국**이었던 것.

- Vercel 함수: `iad1`(미국 워싱턴DC)에서 실행
- Supabase DB: `icn1`(한국 인천/서울)
- → 서버가 DB 쿼리 할 때마다 **태평양 왕복** → options 조회 하나가 warm 상태에도 **1~5초**, 편차 심함

증거(응답 헤더):
- 함수: `X-Vercel-Id: icn1::iad1::...` (요청은 서울로 들어오나 실행은 미국)
- DB: `CF-RAY: ...-ICN` (한국)

**클라이언트(브라우저, 한국)→Supabase는 빨랐다(160ms).** 느린 구간은 오직 **미국 함수 ↔ 한국 DB** 뿐이었다.

### 해결
`vercel.json`에 `"regions": ["icn1"]` — 함수를 Supabase와 같은 서울 리전에서 실행.

| | 이전(미국 함수) | 이후(서울 함수) |
|---|---|---|
| options warm | 1~5초 | **0.28~0.58초** |
| options 첫 cold | ~5초 | ~1.4초 |

함수 리전 확인: `X-Vercel-Id: icn1::icn1::...` (실행도 서울).

## 2. 지문 인증(생체인증) — 반드시 숙지 ⭐

### 동작 구조
1. 로그인 페이지 진입/새로고침 → 등록 기기면 **잠금화면**(BiometricLockScreen) 렌더
2. 렌더 시점에 `prefetchAuthOptions()`가 서버 options를 **미리 요청**(challenge 발급, 쿠키 저장) → 탭하면 즉시 프롬프트
3. 사용자가 **"인증하기" 탭** → `startAuthentication()` → OS 생체 프롬프트
4. 스캔 성공 → verify(서버) → 세션 토큰 + profile 반환 → `setSession` → `window.location.href=/home`

### 오늘 지문 관련 변경
- **자동 실행 제거**: 예전엔 잠금화면 뜨자마자 지문 프롬프트가 자동으로 튀어나옴 → 새로고침만 해도 떠서 혼란. 이제 **"인증하기"를 눌러야** 프롬프트가 뜬다. (`login/page.tsx`의 auto-trigger `useEffect` 제거)
- **"인증하기" 버튼 깜빡임 제거**: 지문 성공 시 `finally { setLoading(false) }`가 실행되며 홈 이동 직전 버튼이 순간 노출됐음. 성공 경로에선 loading을 유지(어차피 홈으로 이동), 실패·취소 경로에서만 `setLoading(false)`. (`useWebAuthn.ts`)
- **challenge 쿠키 경로**: `/api/auth/webauthn/authenticate`, TTL 300초, HMAC 서명. options·verify가 이 쿠키로 challenge 검증. (건드리지 말 것)

### ⚠️ 건드리면 안 되는 구조 (재발 방지)
1. `prefetchAuthOptions` (잠금화면 렌더 시 options 미리 요청)
2. 지문 성공 후 `window.location.href`로 홈 이동 (router.push 아님 — AuthContext 상태 꼬임 방지)
3. challenge 쿠키 경로/서명
4. 성공 시 loading 유지 / 실패·취소 시에만 해제
- 상세: `docs/worklog/작업지시서.md`

### 미해결: 간헐 실패(≈20%)
- 증상: 바로 인증해도 10번 중 2번쯤 "지문 인증에 실패했습니다" → 재시도하면 됨.
- DB 확인: credential 중복 없음(사용자당 1개). → 중복 가설 기각.
- 조치: verify 실패 지점마다 서버 로그(`[wa-verify] FAIL: ...`) 심어둠 **(임시, 원인 확정 후 제거)**.
- **다음 단계**: 실패 재현 시 Vercel 로그를 읽어 어느 단계(쿠키 없음/challenge 만료/서명 검증 실패/세션 발급 실패)인지 확정 → 근본 수정.

## 3. Face ID는? (질문 답변)

**별도 코드 불필요 — 지문과 완전히 같은 흐름.**

- WebAuthn 플랫폼 인증기는 **OS가 생체 방식을 결정**한다. iPhone(Face ID 기기)에서 `startAuthentication()`을 부르면 OS가 **자동으로 Face ID 프롬프트**를 띄운다. 안드로이드면 지문, Touch ID 맥이면 지문. 우리 코드는 동일.
- `BiometricLockScreen`의 `detectBiometricType()`는 iPhone이면 "Face ID로 로그인하세요" + Face ID 아이콘, 그 외엔 "지문으로 로그인하세요"로 **라벨·아이콘만** 바꾼다. 실제 인증 로직엔 분기 없음.
- 카운터: iOS 패스키(iCloud 동기화)는 counter가 항상 0으로 오는 경우가 많은데, verify(`@simplewebauthn`)가 0/0을 정상 허용하므로 문제없음. (DB의 counter=49 credential은 값이 증가하는 안드로이드류 인증기)
- 전제: iOS 16.4+ (그 이하 Safari는 JS 미실행 이슈 — 별도 해결됨, 메모리 `ios-todo` 참고).

즉 Face ID 기기 사용자는 "인증하기" 탭 → **Face ID로 즉시 로그인**. 추가 작업 없음.

## 4. 홈 화면 로딩 개선

- **홈 카드 "불러오는중" 제거**: 기안(`useMyDrafts`)·결재(`useApprovalInbox`) 카드가 프로필 로드 후 각자 Supabase를 조회하며 "불러오는중"을 띄웠음. **stale-while-revalidate 캐시**(`src/lib/local-cache.ts`) 도입 — 지난 결과를 localStorage에 저장, 재진입 시 즉시 표시 + 백그라운드 갱신. 사용자별 캐시 키.
- **공지 캐시**: `AnnouncementSection`도 동일하게 localStorage 캐시로 즉시 표시.
- **홈 shell CDN 정적 서빙**: `(main)/layout.tsx`의 `export const dynamic = "force-dynamic"` 제거 → `/home` 등이 `○ (Static)`으로 빌드돼 CDN 즉시 서빙.
- **프로필 캐시(기존 유지)**: 생체 verify 응답에 profile 포함 → sessionStorage 캐시 → 홈 진입 시 Supabase 재조회 생략.

- **로그아웃 즉시 처리**: `signOut`이 `await supabase.auth.signOut()` 네트워크를 기다려 스피너가 돌았음. `signOut({ scope: "local" })`(네트워크 대기 없음) + `window.location.href="/login"` 하드 리다이렉트로 **즉시** 이탈. (`AuthContext.tsx`)

- **소망·현대는 같은 코드베이스 + 같은 Supabase DB(`bnlybtvdqyfxmbrcvcnv`)** 공유. 법인 구분은 `NEXT_PUBLIC_CORP`(SM/HD)와 **별도 Vercel 프로젝트**로만.

## 7. cold start 완전 차단 — 함수 예열(warmup)

첫 접속(오래 조용한 뒤 첫 로그인) cold start ~1.4초를 없애기 위해 **외부 모니터가 5분마다 함수를 핑**해 warm 유지.

- 코드: `authenticate/route.ts`에 **GET 핸들러** 추가 — `{ok:true, warm:true}` 반환, DB 미호출. (기존 POST `{action:"ping"}`도 유지)
  - GET로 만든 이유: UptimeRobot 등 무료 모니터가 GET 방식이라 바로 연결 가능.
- 설정: **UptimeRobot(무료)** 에 HTTP(s) 모니터 2개, 5분 간격:
  - `https://somang-erp.vercel.app/api/auth/webauthn/authenticate`
  - `https://hyundai-erp.vercel.app/api/auth/webauthn/authenticate`
- 원리: Vercel 함수는 ~15분 유휴 시 잠듦 → 5분 핑이면 안 잠듦 → 200명 누구도 cold 미경험.

### 참고: "은행앱처럼" 논의
- cold start·간헐실패는 "네이티브 vs 웹"이 아니라 **서버 왕복** 때문. 네이티브로 바꿔도 서버가 그대로면 동일.
- 은행앱이 즉시인 이유 = 지문이 **기기에 저장된 토큰을 여는 로컬 자물쇠**일 뿐 서버 왕복이 없어서. (웹에서도 구현 가능한 설계 선택)
- 결정: 지금은 **핑(A안)**으로 충분. 은행앱식 "로컬 토큰 언락" 재설계는 나중에 별도로 제대로(설계→구현→양기종 테스트). Supabase 세션은 이미 기기에 persist되므로 재설계 시 그 위에 지문 게이트를 얹는 방향.
